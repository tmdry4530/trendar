# 신생 트렌드 수집 (trend-discovery) — 요구사항

## 1. Overview

### 문제 정의
현재 수집은 검색 결과를 `sort=stars&order=desc`로 정렬해 **역대 스타 상위 30개**만 가져온다. 그 결과:
- 5년 전 10만 스타 레포가 2주 전 2천 스타 레포를 항상 이겨서, **표본(상위 30)이 오래된 거대 레포로 채워진다.**
- 정작 "지금 뜨는" 신생 레포는 상위 30 창에 들어오기도 전에 **잘려나가** 수집조차 안 된다.
- 우리 스냅샷/델타 엔진은 모멘텀을 추적할 수 있지만, **표본에 없는 레포는 추적 대상이 아니다.**

즉 수집 필터(스타순)가 제품 목적(트렌드 발견)과 충돌한다.

### 목표
- 기존 스타순 수집은 유지(거대 레포 베이스라인)하되, **생성 N일(기본 90일) 이내 신생 레포 슬라이스를 병행 수집**해 표본에 진입시킨다.
- 신생 레포의 **생성일**을 저장하고 **속도(스타 ÷ 나이)** 로 "핫함"을 산출한다.
- 대시보드에 **'신생 급상승'** 섹션을 추가해 지금 뜨는 신생 레포를 바로 보여준다.

### Non-goals
- Repos 목록의 새 정렬/필터(신생순) 추가 — 이번엔 대시보드 노출만
- GitHub Trending 페이지 스크래핑 — 공식 Search API만 사용
- 스냅샷 보존 정책·다운샘플 — 별도 기능
- 레포-쿼리 다대다(중간 테이블) — 현행 단일 귀속 유지

### 성공 기준
- 생성 3개월 이내 + 스타가 빠르게 붙는 레포가 수집 표본에 들어오고, 대시보드 '신생 급상승'에 **속도 순으로** 노출된다.
- 같은 레포가 여러 슬라이스/쿼리에 걸려도 한 번의 수집에서 **스냅샷이 1행만** 쌓인다(델타 오염 없음).
- 기존 스타순 수집·Top Movers·언어 분포는 그대로 동작한다(무회귀).

## 2. 요구사항

### R1. 신생 트렌드 슬라이스 수집

**유저스토리:** 사용자로서, 지금 막 뜨는 신생 레포를 발견하기 위해, 역대 스타 상위뿐 아니라 최근 생성된 유망 레포도 수집되기를 원한다.

**인수 조건:**
- R1.1 WHEN 한 검색 조건을 수집하면, THE SYSTEM SHALL 기존 스타순 슬라이스에 더해 `created:>{오늘−윈도우일}` 한정자를 붙인 **신생 슬라이스**를 추가로 조회한다
- R1.2 WHEN 신생 슬라이스를 조회하면, THE SYSTEM SHALL 그 조건의 `query_type`(topic/keyword) 질의를 동일하게 적용하고 `sort=stars`로 정렬한다
- R1.3 WHERE 신생 슬라이스에는, THE SYSTEM SHALL 기존 `ETL_MIN_STARS`가 아닌 **신생 전용 하한(`ETL_TREND_MIN_STARS`, 기본 50)** 을 적용한다 (신생 레포는 스타가 낮아 기존 1000 하한이면 전부 걸러지므로)
- R1.4 IF 신생 슬라이스 조회가 실패하면(API 오류 등), THEN THE SYSTEM SHALL 그 슬라이스만 에러로 기록하고 기존 슬라이스·다음 조건 수집은 계속한다

### R2. 생성일 저장 및 속도 산출

**유저스토리:** 사용자로서, "얼마나 빨리 스타를 모았는지"로 신생 레포를 비교하기 위해, 레포의 생성일과 속도 지표를 원한다.

**인수 조건:**
- R2.1 WHEN 레포를 정규화하면, THE SYSTEM SHALL GitHub 응답의 `created_at`을 `repos.github_created_at`에 저장한다
- R2.2 WHEN 속도를 계산하면, THE SYSTEM SHALL `velocity = stars / max(생성 후 경과일, 1)` 로 산출한다 (0일 나눗셈 방지)
- R2.3 WHERE 기존 users/repos 테이블에 컬럼이 없으면, THE SYSTEM SHALL 부팅 시 멱등 마이그레이션으로 `github_created_at` 컬럼을 추가한다

### R3. 수집 중복 제거 (한 실행 = 레포당 1회 처리)

**유저스토리:** 운영자로서, 델타가 오염되지 않도록, 한 번의 수집에서 같은 레포가 두 번 저장되지 않기를 원한다.

**인수 조건:**
- R3.1 WHEN 한 사용자의 수집에서 여러 슬라이스·조건이 같은 `github_id`를 반환하면, THE SYSTEM SHALL 그 레포를 **한 번만** upsert하고 스냅샷을 **1행만** 적재한다
- R3.2 IF 같은 레포가 여러 조건에 매칭되면, THEN THE SYSTEM SHALL 먼저 매칭된 조건의 `query_id`로 귀속한다 (현행 단일 귀속 유지)

### R4. 대시보드 '신생 급상승' 노출

**유저스토리:** 사용자로서, 지금 뜨는 신생 레포를 한눈에 보기 위해, 대시보드에서 속도 순 목록을 원한다.

**인수 조건:**
- R4.1 WHEN 대시보드를 열면, THE SYSTEM SHALL 생성 후 윈도우일 이내인 레포를 **속도(velocity) 내림차순**으로 정렬한 '신생 급상승' 목록을 표시한다
- R4.2 WHEN 각 신생 레포를 표시하면, THE SYSTEM SHALL 레포명·언어·스타와 함께 **생성 경과("3주 전")** 를 함께 보여준다
- R4.3 IF 신생 레포가 하나도 없으면, THEN THE SYSTEM SHALL 빈 상태 안내를 표시한다(에러 아님)
- R4.4 WHERE 이 목록은, THE SYSTEM SHALL 요청 사용자의 데이터만(`user_id` 스코프) 조회한다

### R5. 설정

**유저스토리:** 운영자로서, 트렌드 기준을 조정하기 위해, 윈도우·하한을 환경변수로 바꾸고 싶다.

**인수 조건:**
- R5.1 WHEN `ETL_TREND_WINDOW_DAYS`(기본 90)를 설정하면, THE SYSTEM SHALL 신생 슬라이스의 `created:` 기준과 대시보드 노출 창에 그 값을 적용한다
- R5.2 WHEN `ETL_TREND_MIN_STARS`(기본 50)를 설정하면, THE SYSTEM SHALL 신생 슬라이스의 스타 하한에 그 값을 적용한다
- R5.3 IF 환경변수가 없거나 비정상 값이면, THEN THE SYSTEM SHALL 기본값(90일 / 50스타)을 사용한다
