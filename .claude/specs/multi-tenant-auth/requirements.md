# 멀티테넌트 전환 + GitHub OAuth 인증 — 요구사항

## 1. Overview

### 문제 정의

AgentRadar는 현재 단일 사용자용 전역 공유 구조다. 인증이 전혀 없어 모든 API가 공개되어 있고, watch query·레포·북마크·메모가 전부 하나의 전역 공간에 섞이며, GitHub API 수집도 운영자의 단일 `GITHUB_TOKEN`(env)에 의존한다. 이 상태로는 여러 사용자에게 서비스를 제공(상용화)할 수 없다 — 누구나 남의 쿼리를 지울 수 있고, 모든 사용자의 수집이 토큰 하나의 레이트리밋(5,000 req/hr)을 나눠 쓰게 된다.

### 목표

- 사용자가 자신의 GitHub 계정으로 로그인해 **자기만의 격리된 공간**에서 watch query·레포·북마크·메모를 관리한다.
- 각 사용자의 수집(ETL)은 **그 사용자의 OAuth 토큰**으로 실행되어 레이트리밋이 사용자별로 분산된다.
- 개인 사용자 = 테넌트. 레포·스냅샷을 포함한 **모든 데이터가 사용자별로 완전 격리**된다.

### Non-goals

- 과금·플랜·결제 연동 (추후 별도 기능)
- 팀/조직 테넌시, 멤버 초대·공유
- 비로그인 방문자용 읽기 전용 데모 (비로그인은 로그인 페이지만)
- 운영자(admin) 콘솔·사용자 관리 화면
- 기존 전역 데이터의 마이그레이션 (리셋하고 새 스키마로 시작)
- GitHub 외 다른 로그인 수단 (이메일/비밀번호 등)

### 성공 기준

- 서로 다른 두 GitHub 계정으로 로그인하면 상대방의 쿼리·레포·북마크가 전혀 보이지 않는다.
- 비로그인 상태에서는 어떤 데이터 API도 호출할 수 없다 (401).
- cron 수집 로그에서 각 사용자의 쿼리가 그 사용자의 토큰으로 수집되었음이 확인된다.
- 한 사용자의 토큰이 무효화되어도 나머지 사용자의 수집은 정상 동작한다.
- Railway 프로덕션 배포에서 위 사항이 모두 동작한다.

## 2. 요구사항

### R1. GitHub OAuth 로그인·로그아웃

**유저스토리:** 방문자로서, 별도 회원가입 없이 내 GitHub 계정으로 AgentRadar를 쓰기 위해, GitHub OAuth 로그인을 원한다.

**인수 조건:**
- R1.1 WHEN 비로그인 사용자가 앱의 어떤 화면에 접근하면, THE SYSTEM SHALL 서비스 소개와 "GitHub로 로그인" 버튼이 있는 로그인 페이지를 표시한다.
- R1.2 WHEN 사용자가 GitHub OAuth 인가를 완료하면, THE SYSTEM SHALL 계정을 생성(최초 로그인)하거나 기존 계정과 매칭한 뒤 세션을 발급하고 대시보드로 리다이렉트한다.
- R1.3 WHEN OAuth 콜백이 실패하면(사용자 거부, state 불일치, GitHub 에러), THE SYSTEM SHALL 에러 안내와 함께 로그인 페이지로 되돌린다.
- R1.4 WHEN 로그인된 사용자가 로그아웃하면, THE SYSTEM SHALL 세션을 무효화하고 로그인 페이지로 보낸다.
- R1.5 WHILE 세션이 유효한 동안, THE SYSTEM SHALL 브라우저 재방문 시 재로그인 없이 로그인 상태를 유지한다.
- R1.6 WHEN OAuth 인가를 요청할 때, THE SYSTEM SHALL 필요한 최소 스코프만 요청한다 (공개 레포 검색 + 사용자 프로필 식별).
- R1.7 WHILE 로그인 상태일 때, THE SYSTEM SHALL 상단 네비게이션에 사용자의 GitHub 아바타·이름과 로그아웃 수단을 표시한다.

### R2. 테넌트 데이터 격리

**유저스토리:** 로그인 사용자로서, 내 데이터가 다른 사용자와 섞이거나 노출되지 않도록, 완전한 사용자별 격리를 원한다.

**인수 조건:**
- R2.1 WHEN 비인증 요청이 데이터 API(`/api/*`)에 도달하면, THE SYSTEM SHALL 401을 반환한다 (인증·헬스체크 엔드포인트 제외).
- R2.2 WHEN 사용자가 watch query를 조회·생성·수정·삭제하면, THE SYSTEM SHALL 본인 소유 쿼리에만 작업을 수행한다.
- R2.3 WHEN 사용자가 레포 목록·상세·스냅샷·통계·트렌드를 조회하면, THE SYSTEM SHALL 본인 쿼리로 수집된 데이터만 반환한다.
- R2.4 WHEN 사용자가 북마크·메모를 변경하면, THE SYSTEM SHALL 본인 레포에만 반영한다.
- R2.5 IF 사용자가 다른 사용자 소유 리소스의 id로 접근하면, THEN THE SYSTEM SHALL 404를 반환한다 (존재 여부를 노출하지 않는다).

### R3. 사용자별 OAuth 토큰으로 ETL 실행

**유저스토리:** 로그인 사용자로서, 내 수집이 다른 사용자와 레이트리밋을 공유하지 않도록, 내 GitHub 토큰으로 내 쿼리가 수집되기를 원한다.

**인수 조건:**
- R3.1 WHEN 사용자가 로그인을 완료하면, THE SYSTEM SHALL 발급된 OAuth 토큰을 암호화하여 저장한다 (평문 저장 금지).
- R3.2 WHEN 주기 cron ETL이 실행되면, THE SYSTEM SHALL 각 사용자의 활성 쿼리를 그 사용자의 저장된 토큰으로 수집한다.
- R3.3 IF 특정 사용자의 토큰이 무효(revoke·만료)하면, THEN THE SYSTEM SHALL 해당 사용자의 수집만 스킵하고 토큰 무효 상태를 기록하며, 나머지 사용자의 수집은 계속한다.
- R3.4 WHILE 토큰이 무효 상태인 동안, THE SYSTEM SHALL 해당 사용자에게 UI에서 재로그인이 필요함을 안내하고, 재로그인 시 새 토큰으로 갱신·상태를 해제한다.
- R3.5 WHEN 사용자가 수동 수집을 실행하면, THE SYSTEM SHALL 본인 쿼리만 본인 토큰으로 즉시 수집한다.
- R3.6 WHEN ETL 상태를 조회하면, THE SYSTEM SHALL 본인 수집의 마지막 실행 시각·결과만 반환한다.

### R4. 남용 방지 기본 상한

**유저스토리:** 운영자로서, 소수 사용자가 수집 자원을 독점하지 못하도록, 사용자당 기본 상한을 원한다.

**인수 조건:**
- R4.1 WHEN 사용자가 상한(env `MAX_QUERIES_PER_USER`, 기본 10)을 초과해 watch query를 생성하려 하면, THE SYSTEM SHALL 생성을 거부하고 상한 안내 에러를 반환한다.
- R4.2 WHILE 상한에 도달한 상태에서도, THE SYSTEM SHALL 기존 쿼리의 조회·수정·삭제는 정상 허용한다.

### R5. 계정 탈퇴

**유저스토리:** 사용자로서, 서비스를 떠날 때 내 데이터가 남지 않도록, 완전 삭제되는 탈퇴 기능을 원한다.

**인수 조건:**
- R5.1 WHEN 사용자가 탈퇴를 확정하면, THE SYSTEM SHALL 그 사용자의 계정·저장된 토큰·쿼리·레포·스냅샷·북마크·메모·세션을 모두 삭제한다.
- R5.2 WHEN 탈퇴한 사용자가 같은 GitHub 계정으로 다시 로그인하면, THE SYSTEM SHALL 빈 상태의 새 계정으로 시작한다.
- R5.3 WHEN 탈퇴 UI를 표시할 때, THE SYSTEM SHALL 되돌릴 수 없음을 명시하고 확인 절차를 거친다.

### R6. 배포 전환

**유저스토리:** 운영자로서, 기존 Railway 배포를 중단 없이 새 구조로 전환하기 위해, 리셋 기반의 명확한 전환 절차를 원한다.

**인수 조건:**
- R6.1 WHEN 새 버전이 배포되면, THE SYSTEM SHALL 기존 전역 데이터를 리셋하고 멀티테넌트 스키마로 초기화한다 (기존 데이터 이관 없음, 시드 쿼리 삽입 없음).
- R6.2 WHEN 프로덕션(Railway)에서 부팅하면, THE SYSTEM SHALL OAuth 콜백 URL·세션·암호화 키 등 필요한 설정을 env로 주입받아 동작한다.
