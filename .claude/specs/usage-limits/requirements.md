# 사용량 한도 (수동 수집 일일 한도) — 요구사항

## 1. Overview

### 문제 정의
수동 수집(`POST /api/etl/run`)에 호출 횟수 제한이 없다. 동시 실행 락(사용자당 1개)만 있어서, 순차적으로는 하루에 수백 번도 호출할 수 있다. 그 결과:
- `repo_snapshots`에 스냅샷이 무의미하게 누적된다 (저장 낭비 + 짧은 간격 스냅샷은 델타의 의미를 왜곡)
- 서버(DB 쓰기·GitHub API 왕복)에 불필요한 부하가 생긴다
- 상용 서비스로서 남용에 대한 방어선이 없다

### 목표
- 사용자당 수동 수집을 **하루 10회**(기본값, env로 조정 가능)로 제한한다. 리셋 기준은 **KST 자정**.
- 남은 횟수를 **수집 버튼 근처에 상시 표시**하고, 소진 시 버튼을 비활성화해 사용자가 누르기 전에 알 수 있게 한다.
- 자동 cron 수집은 한도와 무관하게 그대로 동작한다.

### Non-goals
- 쿼리 CRUD·조회 API에 대한 속도 제한 (쿼리는 이미 개수 상한 10개로 보호됨)
- IP 기반 전역 rate limit (express-rate-limit 등 미들웨어 도입)
- 스냅샷 보존 정책(다운샘플/아카이브) — 별도 기능으로
- 유료 플랜별 차등 한도 — 추후 과금 도입 시

### 성공 기준
- 하루 10회를 소진한 사용자의 11번째 수동 수집이 429로 거부되고, 화면에는 잔여 0회 + 비활성 버튼 + 리셋 안내가 보인다
- KST 자정이 지나면 별도 조치 없이 카운터가 새로 시작된다
- cron 자동 수집은 사용자의 잔여 한도와 무관하게 계속 돈다

## 2. 요구사항

### R1. 수동 수집 일일 한도

**유저스토리:** 운영자로서, 서버·데이터 남용을 막기 위해, 사용자별 수동 수집 횟수를 하루 단위로 제한하고 싶다.

**인수 조건:**
- R1.1 WHEN 사용자가 수동 수집을 요청하고 오늘(KST 기준) 사용 횟수가 한도 미만이면, THE SYSTEM SHALL 수집을 실행하고 사용 횟수를 1 증가시킨다
- R1.2 IF 오늘 사용 횟수가 한도에 도달한 상태에서 수동 수집을 요청하면, THEN THE SYSTEM SHALL 수집을 실행하지 않고 429 `ETL_DAILY_LIMIT_EXCEEDED` 와 리셋 시각 정보를 반환한다
- R1.3 WHEN KST 자정이 지나면, THE SYSTEM SHALL 별도 배치 작업 없이 다음 요청부터 사용 횟수를 0에서 다시 센다
- R1.4 IF 한도 검사는 통과했으나 수집이 시작 전 단계에서 거부되면(중복 실행 409, 토큰 무효 409), THEN THE SYSTEM SHALL 사용 횟수를 증가시키지 않는다
- R1.5 WHILE cron 자동 수집이 실행되는 동안, THE SYSTEM SHALL 사용자의 수동 한도를 소비하거나 검사하지 않는다
- R1.6 WHEN 환경변수로 한도를 조정하면(`MAX_MANUAL_ETL_PER_DAY`, 기본 10), THE SYSTEM SHALL 재시작 후 새 한도를 적용한다
- R1.7 IF 서버가 재시작되어도, THEN THE SYSTEM SHALL 오늘 사용 횟수를 유지한다 (인메모리가 아닌 영속 저장)

### R2. 잔여 한도 화면 표시

**유저스토리:** 사용자로서, 수집 버튼을 누르기 전에 오늘 몇 번 남았는지 알기 위해, 잔여 한도를 화면에서 보고 싶다.

**인수 조건:**
- R2.1 WHEN 수집 버튼이 있는 화면이 로드되면, THE SYSTEM SHALL 버튼 근처에 "오늘 N/한도회" 형식의 잔여 정보를 표시한다
- R2.2 WHEN 수동 수집이 성공적으로 시작되면, THE SYSTEM SHALL 화면의 잔여 횟수를 갱신한다
- R2.3 IF 잔여 횟수가 0이면, THEN THE SYSTEM SHALL 수집 버튼을 비활성화하고 리셋 시각(KST 자정)을 안내한다
- R2.4 WHEN 사용자가 429 응답을 받으면(다른 탭 사용 등으로 표시가 뒤처진 경우), THE SYSTEM SHALL 에러 메시지를 토스트로 안내하고 잔여 표시를 0으로 동기화한다

### R3. 한도 상태 API

**유저스토리:** 프론트엔드로서, 잔여 한도를 그리기 위해, 수집 상태 API에서 한도 정보를 함께 받고 싶다.

**인수 조건:**
- R3.1 WHEN `GET /api/etl/status`를 호출하면, THE SYSTEM SHALL 기존 응답에 오늘 사용 횟수·한도·잔여 횟수를 포함해 반환한다
- R3.2 WHEN 날짜가 바뀐 뒤 처음 status를 조회하면, THE SYSTEM SHALL 사용 횟수 0 기준의 값을 반환한다 (리셋이 조회에도 반영)
