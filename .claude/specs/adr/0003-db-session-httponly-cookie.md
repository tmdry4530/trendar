# ADR-0003: 세션 — DB 세션 + httpOnly 쿠키 (JWT 아님)

- **날짜**: 2026-07-13
- **상태**: 승인됨
- **관련 기능**: multi-tenant-auth

## 맥락

로그인 세션 유지 방식이 필요하다. 프로덕션에서 백엔드가 프론트를 같은 오리진으로 통합 서빙하므로(단일 컨테이너) 쿠키 기반이 자연스럽고, 계정 탈퇴(R5.1)·로그아웃(R1.4) 시 세션 즉시 무효화가 요구된다.

## 검토한 대안

1. **DB 세션 + httpOnly 쿠키** — 불투명 랜덤 토큰을 `sessions` 테이블에 해시로 저장, httpOnly·secure·SameSite=Lax 쿠키로 전달. 장점: 로그아웃·탈퇴 시 행 삭제로 즉시 무효화, XSS로 토큰 접근 불가, Node `crypto`만으로 구현(의존성 0). 단점: 요청마다 세션 조회 1회.
2. **JWT httpOnly 쿠키** — 서명된 JWT를 쿠키에 담아 stateless로. 장점: DB 조회 없음. 단점: 즉시 무효화 불가(만료까지 유효) → R1.4·R5.1과 충돌, 블랙리스트를 두면 결국 stateful, `jsonwebtoken` 의존성 추가.

## 결정

**DB 세션 + httpOnly 쿠키**를 선택했다. 즉시 무효화 요구사항이 있고, 이 서비스 규모에서 요청당 세션 조회 1회는 무시 가능한 비용이다. 세션 토큰은 SHA-256 해시로 저장해 DB 유출 시에도 세션 하이재킹이 불가능하게 한다.

## 결과

- `sessions` 테이블이 추가되고 인증 미들웨어가 요청마다 세션→사용자를 조회한다.
- 트래픽이 커지면 세션 조회 캐시(메모리/Redis)를 덧붙일 수 있는 구조로 남는다.
- CSRF는 SameSite=Lax 쿠키 + 상태 변경 API가 전부 JSON body 기반(form POST 아님)인 점으로 방어하되, OAuth 콜백은 `state` 파라미터로 별도 방어한다.
