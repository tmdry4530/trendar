# ADR-0005: 트렌드 표본 — 병행 슬라이스 수집 + velocity 조회 시 계산

- **날짜**: 2026-07-19
- **상태**: 승인됨
- **관련 기능**: trend-discovery

## 맥락

수집이 `sort=stars` 상위 30개만 가져와 표본이 오래된 거대 레포로 채워지고, 신생 급상승 레포는 수집조차 되지 않았다. "지금 뜨는 레포"를 표본에 넣는 방법과, 신생 레포의 "핫함"을 산출·저장하는 방식의 결정이 필요했다.

## 검토한 대안

### 표본 확보
1. **병행 슬라이스** — 기존 스타순 검색에 더해 `created:>N일전` 슬라이스를 추가 조회. 장점: 거대 레포 베이스라인 유지, 변경 국소적(질의 1개 추가), 공식 API 문법만 사용. 단점: 쿼리당 API 호출 2배(10조건 기준 20회/수집 — 분당 30회 한도 내).
2. **정렬 교체(updated/최근성 순)** — 기존 검색의 정렬만 변경. 장점: 호출 수 불변. 단점: 거대 레포 추적이 사라져 기존 화면(Top Movers 베이스라인) 회귀, "최근 push"는 "핫함"과 다른 신호.
3. **GitHub Trending 스크래핑** — 장점: 큐레이션 품질. 단점: 비공식(파손 리스크), 키워드 필터 불가 — 제품의 "내 니치" 컨셉과 불일치.

### velocity 산출
1. **조회 시 SQL 계산** — `stars / GREATEST(DATEDIFF(NOW(), created_at), 1)`. 장점: 나이가 매일 변해도 항상 현재 기준, 컬럼·갱신 로직 불필요. 단점: 정렬이 인덱스를 못 타지만 사용자당 행 수(수백)에선 무의미.
2. **저장 컬럼 + 수집 시 갱신** — 장점: 인덱스 정렬. 단점: 수집(6시간) 사이에 값이 낡고, 갱신 누락 버그 표면 추가.

## 결정

**병행 슬라이스 + 조회 시 계산**을 선택했다. 슬라이스별로 스타 하한을 분리(`ETL_MIN_STARS` 1000 / `ETL_TREND_MIN_STARS` 50)해 신생 슬라이스가 하한에 전멸하지 않게 하고, 실행 단위 dedup(Set)으로 슬라이스·조건 간 같은 레포의 스냅샷 중복 적재를 차단한다.

## 결과

- API 호출이 쿼리당 2회가 되지만 조건 상한(10개) 덕에 Search 한도(30/min) 내에 유지된다.
- 신생 레포는 첫 수집부터 velocity로 순위가 매겨진다(델타 없이도 의미 있는 랭킹).
- dedup 도입으로 기존의 "같은 레포가 두 쿼리에 걸리면 스냅샷 2행" 잠재 버그도 함께 해소된다.
- 사용자·데이터가 크게 늘어 velocity 정렬이 느려지면 저장 컬럼+인덱스로 전환을 재검토한다.
