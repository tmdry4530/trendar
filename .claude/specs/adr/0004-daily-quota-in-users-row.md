# ADR-0004: 일일 한도 카운터 — users 행 + 조건부 원자 UPDATE

- **날짜**: 2026-07-15
- **상태**: 승인됨
- **관련 기능**: usage-limits

## 맥락

수동 수집을 사용자당 하루 10회로 제한하려면 "누가 오늘 몇 번 썼는지"를 저장하고, 동시 요청에도 한도를 넘기지 않게 세야 한다. 저장 위치와 카운팅 방식의 결정이 필요했다.

## 검토한 대안

1. **users 행에 컬럼 2개 (`manual_etl_date`, `manual_etl_count`) + 조건부 원자 UPDATE** — 장점: 사용자당 1행이라 조인·정리(청소) 불필요, 검사+증가가 UPDATE 한 문장이라 동시성 틈 없음, 리셋은 "날짜 비교"로 대체되어 배치 불필요, 의존성 0. 단점: "언제 몇 회 썼는지" 이력이 안 남음, 한도 종류가 늘면 컬럼도 늘어남.
2. **별도 사용 이력 테이블 (usage_events append + COUNT)** — 장점: 감사 로그·통계 가능, 한도 종류 확장 유연. 단점: 매 요청 INSERT+COUNT(또는 집계 테이블 동기화), 오래된 행 청소 배치 필요, 동시성은 결국 별도 처리 필요. 현재 요구(횟수 제한 하나)에 과함.
3. **인메모리/Redis 카운터** — 장점: 빠름. 단점: 인메모리는 재시작 시 리셋(R1.7 위반), Redis는 인프라 의존성 추가 — 단일 인스턴스 규모에 과함.

## 결정

**대안 1**을 선택했다. 요구사항이 "하루 단위 횟수 제한" 하나뿐이고, MySQL 단일 UPDATE 문의 원자성만으로 race condition 없이 구현된다. 이력이 필요해지는 시점(과금·통계)에 대안 2로 확장한다.

## 결과

- 리셋 배치·청소 작업·추가 인프라가 없다. 실패 지점이 UPDATE 한 문장으로 수렴한다.
- 사용 이력·시간대별 통계는 포기한다 — 필요해지면 usage_events 테이블을 추가하고 카운터는 캐시로 강등하는 마이그레이션 경로가 있다.
- 한도 종류가 늘어나면(예: API 호출 한도) 컬럼 증가 대신 그 시점에 구조 재검토.
