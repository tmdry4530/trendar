# ADR-0001: 테넌트 데이터 모델 — 완전 사용자별 격리

- **날짜**: 2026-07-13
- **상태**: 승인됨
- **관련 기능**: multi-tenant-auth

## 맥락

멀티테넌트 전환 시 repo·스냅샷 데이터를 테넌트 간에 어떻게 다룰지 결정이 필요했다. 현재 `repos.github_id`는 전역 UNIQUE이고 북마크·메모가 `repos` 행의 컬럼에 박혀 있어, 어느 쪽을 택하든 스키마 재설계가 필요하다.

## 검토한 대안

1. **전역 공유 풀 + 사용자별 상태 분리** — repo·스냅샷은 전역 1벌, 쿼리↔repo 연결과 북마크·메모만 사용자별 조인 테이블로 분리. 장점: 같은 레포를 N명이 watch해도 수집·저장 1회(API 호출 절약). 단점: 스키마 복잡도 상승(조인 테이블 2개 추가), "누구 토큰으로 이 레포를 수집하나" 소유권이 모호해짐, 쿼리·통계 전부 조인 필요.
2. **완전 사용자별 격리** — repo·스냅샷까지 사용자마다 별도 행. `UNIQUE(user_id, github_id)`. 장점: 격리가 자명하고 모든 쿼리가 `WHERE user_id = ?` 한 줄, 기존 `repos.note`/`is_bookmarked` 컬럼 구조를 그대로 유지, 계정 탈퇴가 CASCADE 삭제 하나로 끝남. 단점: 같은 레포를 N명이 watch하면 N배 수집·저장.

## 결정

**완전 사용자별 격리**를 선택했다. 수집이 사용자별 토큰으로 실행되므로(각자의 5,000 req/hr) 중복 수집이 서비스 전체 레이트리밋을 소모하지 않고, 초기 사용자 규모에서 저장량 중복은 무시 가능하다. 격리·탈퇴·권한 검사의 단순함이 저장 효율보다 중요하다고 판단했다.

## 결과

- 모든 데이터 접근이 `user_id` 스코프 필터 하나로 통제되어 격리 버그 여지가 최소화된다.
- 사용자 수가 크게 늘어 저장량·수집 중복이 문제가 되면 공유 풀 구조로 마이그레이션해야 하며, 그 비용을 감수한다.
- `repos` 테이블의 `github_id` 전역 UNIQUE는 `UNIQUE(user_id, github_id)`로 바뀐다.
